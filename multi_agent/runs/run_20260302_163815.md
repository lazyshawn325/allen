2. 锁机制验收：确认 `multi_agent/runs/v5.lock` 运行后被释放。

P2（增强项）

3. 形成标准化 smoke 报告模板（通过/阻断/修复优先级）。

C. 今日可直接执行的3步  
1. 环境与核心链路  
```bash
cd /home/ubuntu/.openclaw/workspace

command -v codex || echo "codex missing"

  --agents codex,gemini \
  --plan --synthesize --synth-agent codex \
  --retries 0 \
  --out multi_agent/runs/smoke_core.md \
  --save-json multi_agent/runs/smoke_core.json
```

2. v5 主链路（静默）  
```bash
python3 multi_agent/v5_runner.py \
  --task "v5 smoke: runner端到端检查" \
  --notify-mode silent
```

3. 验收与失败注入  
```bash
ls -lt multi_agent/runs | head -n 20
test -f multi_agent/runs/v5.lock && echo "lock still exists (abnormal)" || echo "lock released (ok)"

python3 multi_agent/orchestrator.py "v5 smoke: fail case" \
  --agents not_exist_agent \
  --out multi_agent/runs/smoke_fail.md || true
```

D. 风险与回退方案  
1. 命令缺失（`codex/gemini` 不存在）  
回退：降级单 agent，先完成 `orchestrator` 本地链路验收。  

2. `v5.lock` 残留导致后续任务跳过  
回退：确认无运行中进程后清理锁，再重跑一次主链路。  

3. 推送/外部依赖导致不稳定（token/网络）  
回退：smoke 全程禁推送、静默通知；通过后再单测推送路径。  

4. 通知误报或漏报  
回退：smoke 固定 `silent`，发布前切 `fail-only` 并做一次失败注入复验。  

5. 工件不完整但流程返回成功（假阳性）  
回退：以工件完整性和锁释放作为硬门槛，不满足即阻断放行。
